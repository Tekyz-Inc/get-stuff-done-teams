## Debug Ledger Schema (.gsd-t/debug-state.jsonl)

Each line is a JSON object representing one debug iteration:

```json
{
  "iteration": 1,
  "timestamp": "2026-03-24T15:30:00.000Z",
  "test": "auth.spec.ts:45",
  "error": "timeout waiting for selector #login-btn",
  "hypothesis": "selector changed after refactor",
  "fix": "updated selector to [data-testid=login]",
  "fixFiles": ["src/components/LoginForm.tsx"],
  "result": "STILL_FAILS",
  "learning": "selector is correct — problem is upstream, component never mounts",
  "model": "sonnet",
  "duration": 45
}
```

### Required Fields

| Field       | Type     | Description |
|-------------|----------|-------------|
| iteration   | number   | Sequential iteration number (1-based) |
| timestamp   | string   | ISO 8601 timestamp |
| test        | string   | Test identifier (file:line or test name) |
| error       | string   | Error message or failure description |
| hypothesis  | string   | What the fix attempt is testing |
| fix         | string   | Description of the change made |
| fixFiles    | string[] | Files modified in this iteration |
| result      | string   | "PASS" or "STILL_FAILS" |
| learning    | string   | What was learned regardless of result |
| model       | string   | Model used: "sonnet" or "opus" |
| duration    | number   | Seconds elapsed for this iteration |

### Optional Fields

| Field       | Type     | Description |
|-------------|----------|-------------|
| compacted   | boolean  | true if this entry is from a compacted summary |
| relatedTests | string[] | Other tests affected by the same root cause |

## Ledger API (bin/debug-ledger.js)

### Exports

| Function                        | Signature                                        | Description |
|---------------------------------|--------------------------------------------------|-------------|
| readLedger                      | (projectDir: string) => object[]                 | Read all entries from debug-state.jsonl |
| appendEntry                     | (projectDir: string, entry: object) => void      | Append one entry to the ledger |
| compactLedger                   | (projectDir: string, summary: string) => void    | Replace entries with summary + last 5 raw entries |
| generateAntiRepetitionPreamble  | (projectDir: string) => string                   | Generate prompt preamble listing failed hypotheses and current narrowing direction |
| getLedgerStats                  | (projectDir: string) => object                   | Return { entryCount, sizeBytes, needsCompaction, failedHypotheses, passCount, failCount } |
| clearLedger                     | (projectDir: string) => void                     | Delete the ledger file (called when all tests pass) |

### File Location
- Ledger: `{projectDir}/.gsd-t/debug-state.jsonl`
- Compacted entries have `"compacted": true` flag

## Anti-Repetition Preamble Format

Generated by `generateAntiRepetitionPreamble()` and injected into each `claude -p` session prompt:

```
## Debug Ledger Context (DO NOT retry failed approaches)

### Failed Hypotheses (DO NOT retry these):
1. [iteration 1] "selector changed after refactor" — FAILED: selector is correct, component never mounts
2. [iteration 2] "auth provider not initialized" — FAILED: provider present but token refresh throws

### Current Narrowing Direction:
Based on {N} iterations, the evidence points to: {last learning entry}

### What Has NOT Been Tried:
- {inferred from gap between hypotheses and known component surface area}

### Tests Still Failing:
- {test}: {latest error}
```

## Ledger Compaction Protocol

**Trigger**: `getLedgerStats().sizeBytes > 51200` (50KB)

**Process**:
1. Headless loop controller detects threshold via `getLedgerStats().needsCompaction`
2. Spawns a haiku session: "Read this debug ledger. Produce a condensed summary: tests still failing and why, root causes confirmed, hypotheses eliminated (grouped), most promising unexplored direction."
3. Calls `compactLedger(projectDir, summary)` which:
   - Writes one entry with `"compacted": true` containing the summary
   - Preserves the last 5 raw entries unchanged
   - Replaces all other entries

## Headless Debug-Loop CLI Interface

### Invocation

```
gsd-t headless --debug-loop [flags]
```

### Flags

| Flag               | Default | Description |
|--------------------|---------|-------------|
| --max-iterations   | 20      | Hard ceiling on iterations |
| --test-cmd         | (auto)  | Override test command (default: auto-detect from project) |
| --fix-scope        | (all)   | Limit fix scope to specific files or test patterns |
| --json             | false   | Output structured JSON after each iteration |
| --log              | false   | Write per-iteration logs to .gsd-t/ |

### Escalation Tiers

| Iterations | Model  | Behavior |
|------------|--------|----------|
| 1-5        | sonnet | Standard debug — one fix per session |
| 6-15       | opus   | Deeper reasoning — reads full ledger, may attempt multi-file fixes |
| 16-20      | STOP   | Write full diagnostic summary, present to user, exit code 4 |

### Exit Codes

| Code | Meaning |
|------|---------|
| 0    | All tests pass |
| 1    | Max iterations reached, tests still failing |
| 2    | Ledger compaction error |
| 3    | Process error (claude CLI not found, timeout) |
| 4    | Escalation stop — needs human decision |

### Iteration Cycle (per iteration)

1. Read ledger via `readLedger()`
2. Check `getLedgerStats().needsCompaction` — if true, run compaction first
3. Generate preamble via `generateAntiRepetitionPreamble()`
4. Select model based on escalation tier
5. Build prompt: preamble + "Fix the failing test(s). Write your fix, then run the test suite. Report results."
6. Spawn `claude -p "{prompt}"` with appropriate --model flag
7. Parse output for test results (pass/fail) and fix description
8. Call `appendEntry()` with iteration results
9. If all tests pass → `clearLedger()`, exit 0
10. If iteration >= max → exit 1 (or 4 if in escalation stop tier)
11. Next iteration

## Command Integration Pattern

Commands that currently have in-context fix-retest loops should delegate to headless debug-loop when:
- A test failure persists after the first in-context fix attempt (the "up to 2 attempts" rule)
- The second attempt also fails

**Delegation pattern** (for command files):
```
If fix attempt 1 fails AND fix attempt 2 fails:
  1. Write current failure context to .gsd-t/debug-state.jsonl via appendEntry
  2. Log: "Delegating to headless debug-loop (2 in-context attempts exhausted)"
  3. Run: gsd-t headless --debug-loop --max-iterations 15
  4. Check exit code:
     - 0: Tests pass, continue with task
     - 1/4: Report failure, log to deferred-items.md
     - 3: Report error
```

This preserves the existing "try twice in-context" behavior for quick fixes while escalating persistent failures to the external loop.
