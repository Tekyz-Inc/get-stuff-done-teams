## Design Addendum — In-Session Launch + Watch Loop (confirmed 2026-04-15)

User confirmed: M36 must support launching from **within** a Claude session (after interactive PRD/feature/milestone setup), then detaching and running a **polling watch loop** in the same session to display live progress. Context exhaustion of the watching session is acceptable — `/compact` or `/clear` + `/user:gsd-t-resume` must transparently re-attach to the still-running supervisor.

**Shape**:
- Supervisor = detached OS process (`nohup node bin/gsd-t-unattended.js ... &`, PID written to `.gsd-t/.unattended/supervisor.pid`). Fire-and-forget. In-session Claude does NOT hold the bash tool open.
- Watch loop = new slash command `/user:gsd-t-unattended-watch` fired via `ScheduleWakeup` (dynamic /loop) every **270 seconds** (cache-friendly — 300s would blow the 5-minute prompt cache TTL on every tick). Each tick is fully self-contained: reads state files, displays status, decides whether to reschedule. Zero state between firings.
- Launch command = new slash command `/user:gsd-t-unattended` — pre-flights, spawns detached supervisor, verifies liveness, displays initial status block, schedules first watch tick, returns.

**Confirmed parameters** (user agreed 2026-04-15):
1. **Tick cadence = 270s** (stays in prompt cache; ~320 ticks over a 24h run).
2. **Watch loop ships in Phase 1**, not Phase 4/5. Rationale: without it, the UX is "here's a PID, good luck" — the exact detached-server footgun the user has flagged before (feedback_detached_servers memory). The watch loop is what makes detaching feel supervised.

**Watch tick display (v1 — minimal)**:
```
⏱ gsd-t unattended — elapsed {hms} | iter {n}/{max}
📍 {milestone} · wave {w} · task {t}/{total}
🔄 Last worker: exit {code} ({ago})
📊 Tokens this run: {k} | Est. remaining: {hms}
─────
{last 3 meaningful lines from run.log}
```
Liveness + progress + last log line only. Health (gutter warnings, error counts) and cost detail land in v2.

**Terminal conditions** (each tick checks, in order):
1. `state.status === "done"` → print final report, STOP rescheduling.
2. `state.status === "failed"` → print failure + log path, STOP.
3. `state.status === "stopped"` → user-requested stop via sentinel file, STOP.
4. `kill -0 $PID` fails → supervisor crashed, print diagnostics, STOP.
5. else → `ScheduleWakeup(270, "/user:gsd-t-unattended-watch")` and end turn.

**Resume integration (MANDATORY — Phase 1)**:
`commands/gsd-t-resume.md` Step 0 must check `.gsd-t/.unattended/supervisor.pid`. If it exists AND `kill -0` succeeds AND `state.status` is not a terminal value, resume SKIPS normal resume logic and jumps straight to `/user:gsd-t-unattended-watch`. This is what makes `/clear` + resume a no-op for the watching session — the supervisor is the source of truth, the in-session conversation is disposable.

**Stop mechanism**: sentinel file `.gsd-t/.unattended/stop`. The supervisor loop checks for it between workers. Survives terminal closes. Race-free. Exposed via new `/user:gsd-t-unattended-stop` slash command (trivial — just `touch`es the sentinel).
