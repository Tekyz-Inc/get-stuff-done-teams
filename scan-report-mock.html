Key Findings
🛑

Security — Immediate Action Required

Two critical issues before next release: (1) id_token from Google OAuth accepted without JWKS verification — enables token forgery. (2) Raw SQL at tasks.repository.ts:211 vulnerable to injection. Both are 1-day fixes.

⚡

Architecture — Notifications Domain Needs Decoupling

NotifyService directly imports TasksRepository — a hidden cross-domain dependency. Replace with a domain event bus (BullMQ event queue or NestJS EventEmitter). Effort: M (1–2 days).

📊

Reliability — Silent Job Failures

No dead-letter queue for notification jobs. Failed email/push deliveries are silently dropped. Add BullMQ DLQ and failure alerting (Sentry or Slack webhook on repeated failure).

📄

Quality — Test Coverage Gap

Overall coverage 61% (target 80%). Notifications domain critically under-tested at 18%. Recommend a dedicated milestone to bring Notifications to 80% coverage before adding new features.

✅

Strengths — Well-Structured Core

Auth and Projects domains are healthy (88% and 82%). Layered Controller → Guard → Service → Repository pattern consistently applied except in Notifications. BullMQ async architecture is a solid foundation.
